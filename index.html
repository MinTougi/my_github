Dominando o Essencial: Git & GitHub

Comandos Essenciais

git init                  # Inicia um novo repositório local
git clone <url>           # Clona um projeto remoto
git status                # Checa o estado atual das mudanças
git add <arquivos>        # Adiciona arquivos para o próximo commit
git commit -m "mensagem"  # Salva as mudanças
git switch -c feature/x   # Cria e muda para uma nova branch
git push -u origin feature/x # Envia a branch para o remoto
git pull --rebase origin main # Atualiza sua branch com as novidades

Glossário de Termos

Git vs. GitHub
Git é um sistema de controle de versão local. GitHub é a plataforma online para hospedar seus projetos Git.
Repositório (repo)
A pasta do seu projeto, versionada pelo Git.
Commit
Um "instantâneo" das suas mudanças, sempre com uma mensagem descritiva.
Branch
Uma linha de desenvolvimento paralela para novas funcionalidades, como feature/nova-pagina.
Remote
A versão do seu repositório que fica na nuvem, como o origin no GitHub.
Clone
Baixar uma cópia do repositório remoto para sua máquina.
Fork
Criar uma cópia de um repositório no seu próprio perfil do GitHub para poder colaborar.
Pull Request (PR)
Proposta para juntar suas alterações em outra branch, geralmente na main. É o momento de revisão de código.
Stash
Armazenar mudanças temporariamente sem precisar fazer um commit.
Tag/Release
Uma marcação permanente em um ponto específico do histórico, usada para versões como v1.0.0.

Fluxos de Trabalho (Workflows)

GitHub Flow (Simples e Direto)

Ideal para times que precisam de agilidade e entrega contínua.

  1. Parta da branch main e crie uma nova branch para sua tarefa.
  2. Trabalhe e faça commits com mensagens claras.
  3. Envie sua branch para o GitHub.
  4. Abra um Pull Request e descreva suas mudanças.
  5. Aguarde a revisão de código e a aprovação dos testes.
  6. Faça o "Squash & Merge" para a main e delete a branch.

Git Flow (Estrutura Completa)

Ótimo para projetos que precisam de um gerenciamento mais rígido de versões e lançamentos.

  • main: sempre contém o código estável.
  • develop: a branch principal para o desenvolvimento.
  • feature/*: branches para novas funcionalidades.
  • release/* e hotfix/*: branches específicas para preparação de lançamentos e correções urgentes.

Exemplo de um ciclo de trabalho:

# 1) Inicie a partir da develop
git switch develop
git switch -c feature/login

# 2) Trabalhe, commite e envie
# 3) Abra um PR para 'feature/login -> develop'
# 4) Quando pronto, prepare a release e merge em main

Dica: Estabeleça padrões de mensagens de commit e templates de Pull Request. Isso acelera o processo de revisão e melhora a organização do projeto.

Lidando com Conflitos de Merge

Quando surgem? Conflitos acontecem quando duas pessoas alteram a mesma parte do mesmo arquivo. Isso geralmente ocorre em operações de merge ou rebase.

Passo a Passo para Resolver

  1. Execute o comando que causou o conflito, como git pull --rebase origin main.
  2. O Git irá pausar a operação e marcar os arquivos com conflito.
  3. Abra cada arquivo e edite as seções marcadas com <<<<<<< e =======, decidindo qual versão do código você quer manter.
  4. Depois de resolver, adicione o arquivo com git add ..
  5. Finalize a operação com git rebase --continue.
  6. Envie as mudanças para o remoto com git push.

Fica a dica: Sincronize seu trabalho com o repositório remoto com frequência. Isso reduz drasticamente a chance de encontrar conflitos grandes e complexos.
